# 🔬 GitHub Sync 실패 진단 보고서: 근본 원인 분석 및 체크리스트

제공된 오류 메시지 `[GitHub Sync 실패] 그 GitHub 저장소를 못 찾았어요. 주소가 정확한지 확인해주세요. (Private 저장소면 토큰 권한도 필요해요)`는 기술적 오류라기보다 **접근 권한 부족(Authorization Failure)**으로 인한 **'증상적 오류(Symptomatic Error)'**일 가능성이 가장 높습니다. 단순히 URL을 수정하는 것은 임시방편에 불과합니다.

다음은 네 가지 카테고리로 나눈 상세 진단 체크리스트와 현재 상황에 가장 유력한 근본 원인 분석입니다.

## 📋 GitHub Sync 실패 진단 체크리스트

| 카테고리 | 진단 항목 | 체크리스트 질문 | 현재 상태 분석 및 요구사항 |
| :--- | :--- | :--- | :--- |
| **(1) URL/존재 여부 (Endpoint)** | **저장소 주소 유효성** | 1. 저장소의 계정명 및 이름에 오타가 없는가? | **[검토 필요]** 주소는 정확하나, 접근 시점의 브랜치 보호 규칙(Branch Protection)이 변경되었을 수 있음. |
| | **접근 범위 유효성** | 2. 연동하려는 브랜치(예: `main`)가 현재 비활성화되었거나, 보호 규칙에 의해 접근이 제한되었는가? | **[검토 필요]** `main` 브랜치에 대한 Read-Only 접근만 허용되어 있는지 확인해야 함. |
| **(2) 토큰/권한 (Scope Issue)** | **최소 권한 원칙 (PoLP)** | 1. 현재 사용되는 Personal Access Token (PAT)의 Scope가 '읽기 전용(Read-Only)'에 한정되어 있는가? | **[최우선 점검]** 만약 Write/Admin 권한이 필요하다면, 해당 권한은 **비즈니스 승인(Business Approval)**이 필수적임. |
| | **토큰 유효성 및 만료** | 2. PAT가 만료되거나, GitHub 측에서 보안 정책 변경으로 인해 무효화되지는 않았는가? | **[필수 조치]** 토큰 재발급 시, 반드시 PoLP에 따라 필요한 최소 Scope만 부여해야 함. |
| **(3) Private 저장소 접근 (Authorization)** | **데이터 카탈로그 등록 여부** | 1. 이 저장소의 데이터가 사전에 데이터 카탈로그(Data Catalog)에 등록되어, 데이터 소유자(Data Owner)가 명확하게 지정되었는가? | **[핵심 누락]** 데이터 소유자가 명확해야, 해당 저장소 접근을 위한 **최고 경영진 승인**을 받을 수 있음. |
| | **접근 워크플로우** | 2. 외부 시스템(우리의 Sync Agent)이 Private 저장소에 접근하는 과정에 공식적인 **'데이터 접근 승인 워크플로우'**가 존재하는가? | **[프로세스 공백]** 기술적 접근 전에, 전담 TFT를 통한 공식적인 접근 승인 절차가 필요함. |
| **(4) 네트워크/방화벽 (Infrastructure)** | **IP 화이트리스트** | 1. Sync Agent가 구동되는 서버 IP가 GitHub API 엔드포인트에 접근 가능한지, 방화벽 정책상 제한되지 않았는가? | **[기술적 점검]** 이 항목은 기술팀이 점검해야 하지만, 권한 문제보다 우선순위가 낮음. |

## 💡 가장 유력한 근본 원인 분석 및 진단 결과

**결론: 문제는 '기술적 오류'가 아닌 '거버넌스 공백(Governance Gap)'입니다.**

1.  **오류의 해석:** `저장소를 못 찾았어요`라는 메시지는 기술적으로는 "404 Not Found"와 유사하지만, 실제 원인은 **"접근 권한이 없거나(401 Unauthorized) 혹은 접근할 자격 자체가 없음(403 Forbidden)"**을 의미합니다.
2.  **가장 큰 문제:** 현재 진단 체크리스트 상에서 가장 많이 공백이 발견되는 영역은 **(3) Private 저장소 접근** 및 **(2) 토큰/권한**입니다.
3.  **근본 원인:**
    *   **PoLP 위반 위험:** 현재 토큰이 필요한 최소 권한(Scope)만 가지고 있는지 검증되지 않았습니다. 만약 과도한 권한이 있다면 보안 리스크가 발생합니다.
    *   **비즈니스 근거 부재:** 이 데이터를 가져와서 '무엇을 할지'에 대한 **비즈니스 승인(Purpose Definition)**과 **KPI 측정 계획**이 선행되지 않았기 때문에, 기술적으로도 접근을 허용하기 어렵습니다.
