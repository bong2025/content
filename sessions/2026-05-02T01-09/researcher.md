# 📜 콘텐츠 자산 '단일 데이터 출처(SSOT)' 적합성 분석 보고서

**[분석 대상]** GitHub Repository: `bong2025/content`
**[분석 목표]** 콘텐츠 자산이 '단일 데이터 출처(SSOT)' 기준에 부합하는지 검토하고, 스키마/메타데이터 리스크 및 개선 방안 제시.

## 1. SSOT 적합성 검토 요약

| 항목 | 현재 상태 | SSOT 부합도 | 핵심 문제점 |
| :--- | :--- | :--- | :--- |
| **구조화 수준** | 비정형 파일 (Markdown, Drafts) 위주 | **낮음** | 콘텐츠의 핵심 속성(메타데이터)이 파일 내용에 녹아있거나, 개별적으로 관리됨. |
| **스키마 일관성** | 낮은 일관성 | **낮음** | 동일한 개념(예: 주제, 타깃)을 지칭하는 속성이 파일마다 다르게 정의되거나 누락됨. |
| **데이터 무결성** | 출처/최종본/초안 구분이 모호함 | **보통** | 파일명이나 커밋 히스토리를 통해 추적 가능하나, 시스템적으로 강제되지 않아 위험함. |
| **결론** | **현재는 '자산 저장소'일 뿐, '신뢰할 수 있는 데이터 출처'가 아님.** | | 비즈니스 목표(KPI 연동) 달성을 위해서는 구조적 개선이 필수적입니다. |

## 2. 발견된 주요 리스크 (Risk Identification)

### 🔴 A. 스키마 불일치 리스크 (Schema Mismatch Risk)
*   **문제:** 콘텐츠의 핵심 메타데이터(예: 주제, 주요 타깃, 핵심 키워드)가 통일된 필드로 관리되지 않습니다.
    *   *예시:* 어떤 콘텐츠는 `Tags:` 섹션에 키워드를 넣고, 다른 콘텐츠는 본문 내에 키워드를 반복하는 등, 데이터가 비표준화되어 있습니다.
*   **영향:** 콘텐츠 간의 집계(Aggregation)나 검색 기반의 KPI 계산(예: '주제 X' 관련 콘텐츠의 평균 CES 계산)이 불가능하거나, 수동 작업으로만 처리되어 확장성이 제로(Zero)입니다.

### 🟠 B. 메타데이터 부족 리스크 (Insufficient Metadata Risk)
*   **문제:** 콘텐츠의 '수명 주기(Lifecycle)'와 '책임 소재(Accountability)'에 대한 메타데이터가 부족합니다.
    *   **필수 누락 항목:**
        1.  **승인 상태 (Approval Status):** (초안 $\to$ 검토 중 $\to$ 최종 승인 $\to$ 비활성화)의 명확한 상태 플래그 부재.
        2.  **최종 검토 일자 (Last Review Date):** 콘텐츠의 신뢰성(Freshness)을 입증할 근거가 없습니다.
        3.  **KPI 연관 매핑:** 이 콘텐츠가 어떤 KPI(CES, LCR)에 직접 기여하는지 명시된 필드가 없습니다.

## 3. 개선 방안 및 액션 플랜 (Improvement Plan)

SSOT를 구축하고 비즈니스 KPI와 연동하기 위해, **'메타데이터 레이어(Metadata Layer)'**를 최우선으로 도입해야 합니다.

| 우선순위 | 액션 항목 | 세부 실행 내용 | 기대 효과 |
| :--- | :--- | :--- | :--- |
| **P1 (최우선)** | **메타데이터 표준 스키마 정의 및 강제 적용** | 모든 콘텐츠 파일의 최상단에 **YAML Frontmatter** 형식을 도입합니다. (예: `--- title: ... author: ... status: DRAFT ---`) | 데이터 검색 및 분석의 기반을 마련하며, 스키마 일관성을 100% 확보합니다. |
| **P2** | **콘텐츠 라이프사이클(CLC) 워크플로우 구축** | 콘텐츠 상태(Status) 필드를 정의하고, 상태 변경 시 **자동 승인 워크플로우**를 거치도록 시스템 게이트를 설정합니다. | 콘텐츠의 신뢰도(Trust)를 높이고, '누가, 언제, 무엇을' 승인했는지 추적할 수 있게 됩니다. |
| **P3** | **KPI 연동 매핑 필드 추가** | 메타데이터에 `Target_KPI: [CES/LCR/CFS]` 필드를 추가하고, 목표치를 기입하도록 의무화합니다. | 콘텐츠 기획 단계부터 KPI 측정과의 연결고리를 강제하여, 비즈니스 임팩트를 극대화할 수 있습니다. |

---
**요약:** 현재의 GitHub 저장소는 '데이터의 원본 저장소' 역할만 수행합니다. 이를 '신뢰할 수 있는 데이터 출처(SSOT)'로 격상시키려면, **기술적 접근(YAML Frontmatter 등)**과 **프로세스적 접근(승인 워크플로우)**을 동시에 적용하는 거버넌스 구축이 선행되어야 합니다.
